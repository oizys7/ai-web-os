# ai-web-os 实现路线图

这个路线图是基于当前最小 hello world 框架整理出来的下一步行动计划。目标不是一次性做大，而是先把最关键的能力链路做稳，再逐层扩展。

## 总体原则

- 先稳住 ABI、HAL、Wasm、前端、AI 五条边界。
- 先做最小闭环，再做能力扩展。
- 所有新能力都要能回到“可观测、可验证、可替换”这三个标准。
- 目前只使用 `C`、`JavaScript`、`TypeScript`、`Rust`、`Python`，不引入其他语言。

## 阶段 1：把核心链路做实

目标：把 hello world 变成一个真正可演进的最小平台。

任务：

1. 设计 `core` 的最小状态机或调度原型。
2. 将 `abi` 补成稳定的输入、输出、错误码、版本兼容接口。
3. 给 `hal/native` 补齐最小宿主能力，包括日志、时间、内存抽象。
4. 为 `core` 和 `abi` 增加单元测试。
5. 让本机 hello world 从“演示程序”变成“可复用宿主样例”。

验收标准：

- `core` 可以独立运行测试。
- `abi` 有明确版本检查。
- 宿主层可以被替换，而核心行为不变。

## 阶段 2：接入 Rust Wasm 组件

目标：把需要 Wasm 化的模块迁出来，形成可复用组件。

任务：

1. 新建 Rust Wasm 模块骨架。
2. 用和 C ABI 对齐的方式导出初始化、tick、事件、查询接口。
3. 把浏览器数字孪生切换到真实 Wasm 核心，而不是 JS 假实现。
4. 保留 JS/TS 宿主层，用于模拟硬件、事件和可视化。

验收标准：

- 同一套状态逻辑可以在本机和浏览器宿主中运行。
- Wasm 模块不直接依赖浏览器 DOM。
- ABI 在 C 和 Rust 之间保持一致。

## 阶段 3：前端工程化

目标：把浏览器页面升级成可持续维护的前端工程。

任务：

1. 将 `apps/browser-digital-twin` 迁移到 TypeScript 工程结构。
2. 拆出状态面板、事件面板、内存面板、控制面板。
3. 为后续 React 组件化预留边界。
4. 把模拟硬件层和 UI 展示层拆开。

验收标准：

- 页面逻辑可以分层维护。
- UI 与模拟层不互相污染。
- 后续接 React 时不需要重写核心状态逻辑。

## 阶段 4：AI 基础能力

目标：把 AI 变成系统基础设施的一部分。

任务：

1. 用 Python 建立模型调用、prompt 管理、评估脚手架。
2. 定义 AI 服务接口，统一模型、上下文、记忆、工具、审计。
3. 做一个最小 AI 场景，例如日志总结或调度建议。
4. 把 AI 输出当成不可信输入，补上审计和权限控制。

验收标准：

- AI 能力通过统一接口接入，而不是散落在业务代码里。
- AI 结果可记录、可回放、可审计。
- AI 不直接破坏核心确定性逻辑。

## 阶段 5：走向更多宿主

目标：为后续的 native、WASI、kernel、bare-metal 迁移打地基。

任务：

1. 把 HAL 拆成 browser、native、WASI、kernel 等宿主实现。
2. 把能力边界写成清晰的 capability 接口。
3. 为不同目标补构建配置。
4. 为性能关键路径保留 C 和 Rust 的可复用实现。

验收标准：

- 同一核心逻辑可在多个宿主上运行。
- 构建目标可扩展。
- 迁移不会反过来污染核心层。

## 当前建议的下一步

1. 先做阶段 1 的第 1 和第 2 项。
2. 同时把 `core` 的最小状态机接口写出来。
3. 然后再接 Rust Wasm 骨架。
